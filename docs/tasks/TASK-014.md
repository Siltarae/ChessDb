## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 폰이 마지막 행에 도달해도 기물 타입이 변하지 않고 그대로 폰으로 남아있는 상태입니다.
- **이 작업의 책임**: 폰이 마지막 행에 도달했을 때 퀸, 룩, 비숍, 나이트 중 하나로 변신하는 프로모션 로직을 구현합니다.

- **이번 작업에서 하지 않는 것**: `[TASK-060]` (수 실행)에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 프로모션(Promotion) 처리 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🎯 1. 작업 목표

- **최종 상태**: 폰이 8행(White) 또는 1행(Black)에 도착할 때 사용자로부터 선택을 받거나 기본값(퀸)으로 기물 타입을 변경합니다.

- **이번 작업의 최소 결과물**:
  - `packages/shared/src/engines/promotion-engine.ts`
  - `packages/shared/src/engines/promotion-engine.spec.ts`
- **성공 기준 (AC)**:
  - 폰이 보드 끝에 도달했을 때 프로모션이 정확히 발생한다.
  - 퀸뿐만 아니라 나이트, 룩, 비숍으로의 선택 프로모션이 모두 가능하다.
  - 프로모션 직후 보드에는 폰이 사라지고 새 기물이 나타남을 확인했다.

## 🛠️ 3. 상세 기술 사양

- **판정 로직**: 폰의 `to` 인덱스가 `Math.floor(to / 8)` 기준으로 0 또는 7인지 확인.
- **인터페이스 확장**:
  - `Move` 인터페이스에 `promotion?: PieceType` 필드 활용.
  - `executeMove` 호출 시 `promotion` 값이 있으면 해당 타입으로 기물 생성.
- **필수 describe/it 목록**:
  - describe: `executeMove with Promotion`
    - it: `폰이 마지막 행 도달 시 지정된 기물(퀸 등)로 변신해야 한다`
    - it: `프로모션 정보가 누락된 경우의 기본값 처리 또는 에러 처리를 수행한다`

- **핵심 조립/흐름 규칙**:
  - 폰이 보드 끝에 도달했을 때 프로모션이 정확히 발생한다.
  - 퀸뿐만 아니라 나이트, 룩, 비숍으로의 선택 프로모션이 모두 가능하다.
  - 프로모션 직후 보드에는 폰이 사라지고 새 기물이 나타남을 확인했다.

- **데이터 모델 해석**:
  - 프로모션은 마지막 랭크 도달 폰과 승격 기물 선택지를 함께 다룹니다.
  - 출력은 승격 선택지 또는 승격이 적용된 새 상태 정보입니다.

- **외부 의존성**:
  - shared 내부 `PieceType`, `Move` 타입
  - `vitest`

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **권장 네이밍**:
  - 공개 함수/타입 이름: 프로모션(Promotion) 처리 책임이 드러나는 이름
  - 내부 helper 이름: 역할이 바로 드러나는 동사형 또는 조합형 이름
  - 핵심 변수명: 상태와 대상이 분명한 이름
  - 피해야 할 이름: data, item, obj, temp

- **이름별 사용 의도와 적용 시점**:
  - `getPromotionOptions`는 선택 가능한 승격 기물 목록을 뜻할 때 사용합니다.
  - `applyPromotion`은 선택된 기물로 상태를 바꾸는 helper 이름으로 적절합니다.

- **인수 이름 가이드**:
  - `move`, `promotionPieceType`, `state`

- **짧은 예시 골격**:

```ts
const nextState = applyPromotion(state, move, promotionPieceType);
```

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 마지막 랭크가 아닌데 프로모션을 허용하는 경우
  - 왕/폰 같은 허용되지 않은 기물을 승격 대상으로 받아들이는 경우

## ⚖️ 4. 기술 제약 및 규칙

- **작성 원칙**:
  - 전역 규칙을 반복 나열하지 말고, 이번 태스크에서 특히 강조해야 하는 제약만 짧게 적는다.

  - `[TASK-060]` (수 실행)에 연결된 범위는 여기서 닫지 않는다.

- **헌법 정렬 규칙**:
  - `../architecture/project-rules.md`의 네이밍, import/export, 상태 규칙을 그대로 따른다.

- **문서화 규칙**:
