## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 기물은 움직이지만 지금까지 어떤 수들이 두어졌는지에 대한 이력이 화면에 남지 않습니다.
- **이 작업의 책임**: 게임 시작부터 현재까지의 모든 착수 내역을 배열로 관리하고, 우측 패널에 표준 기보 형식으로 나열합니다.

- **이번 작업에서 하지 않는 것**: `[TASK-018]` (되돌리기)에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 수순 목록 기록 및 내역 표시 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🎯 1. 작업 목표

- **최종 상태**: 사용자가 수를 둘 때마다 우측 `MoveList` 영역에 `1. e4 e5 2. Nf3 ...` 형태의 기보가 실시간으로 쌓입니다.

- **이번 작업의 최소 결과물**:
  - `apps/web/src/features/move-history/model/move-history-store.ts`
  - `apps/web/src/widgets/move-history/ui/move-history-panel.tsx`
  - `apps/web/src/pages/notation-input-page.tsx`
- **성공 기준 (AC)**:
  - 모든 착수가 순서대로 정확한 SAN 표기법과 함께 목록에 추가된다.
  - 수 번호(1, 2, 3...)가 백의 수 앞에 정확히 붙는다.
  - 목록 클릭 시 해당 시점의 보드를 보여줄 준비가 되어 있다 (데이터 링크 확인).

## 🛠️ 3. 상세 기술 사양

- **데이터 구조**: `history: MoveHistoryItem[]` (from `TASK-063`).
- **UI 컴포넌트**:
  - `MoveListContainer`: 전체 수순을 감싸는 스크롤 영역.
  - `MoveItem`: 개별 수(Ply)를 렌더링하며, 현재 보고 있는 수를 강조.
- **렌더링 규칙**:
  - 2개 수(백, 흑)를 묶어 하나의 행(Move Number)으로 표시.
  - 수순이 많아질 경우 자동으로 최하단으로 스크롤 이동 (`auto-scroll`).

- **핵심 조립/흐름 규칙**:
  - 모든 착수가 순서대로 정확한 SAN 표기법과 함께 목록에 추가된다.
  - 수 번호(1, 2, 3...)가 백의 수 앞에 정확히 붙는다.
  - 목록 클릭 시 해당 시점의 보드를 보여줄 준비가 되어 있다 (데이터 링크 확인).

- **데이터 모델 해석**:
  - 수순 목록 데이터는 반수 단위 히스토리와 표시용 SAN 문자열 또는 move label을 포함합니다.
  - 출력은 화면에 그릴 목록 항목과 현재 선택 인덱스입니다.

- **외부 의존성**:
  - `react`
  - `zustand`
  - `@chess-db/shared` SAN 또는 move label 데이터

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **권장 네이밍**:
  - 공개 함수/타입 이름: 수순 목록 기록 및 내역 표시 책임이 드러나는 이름
  - 내부 helper 이름: 역할이 바로 드러나는 동사형 또는 조합형 이름
  - 핵심 변수명: 상태와 대상이 분명한 이름
  - 피해야 할 이름: data, item, obj, temp

- **이름별 사용 의도와 적용 시점**:
  - `moveHistoryStore` 또는 `useMoveHistoryStore`는 표시용 수순 히스토리를 관리할 때 사용합니다.
  - `selectedPlyIndex`는 현재 선택된 반수 인덱스를 뜻할 때 유지합니다.

- **인수 이름 가이드**:
  - `historyItems`, `selectedPlyIndex`, `moveLabel`

- **짧은 예시 골격**:

```ts
const { historyItems } = useMoveHistoryStore();
```

- **필수 describe/it 목록**:
  - 최상위 describe: `수순 목록 기록 및 내역 표시`
  - 필수 it: 요구사항 문서의 완료 기준을 직접 검증하는 테스트

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 수순 기록 순서가 실제 착수 순서와 어긋나는 경우
  - 보드 히스토리와 목록 히스토리를 별도 소스로 관리해 불일치가 생기는 경우

## ⚖️ 4. 기술 제약 및 규칙

- **작성 원칙**:
  - 전역 규칙을 반복 나열하지 말고, 이번 태스크에서 특히 강조해야 하는 제약만 짧게 적는다.

  - 현재 디렉토리 구조와 연관 설계 문서의 책임 경계를 유지한다.

- **범위 규칙**:
  - `[TASK-018]` (되돌리기)에 연결된 범위는 여기서 닫지 않는다.

- **헌법 정렬 규칙**:
  - `../architecture/project-rules.md`의 네이밍, import/export, 상태 규칙을 그대로 따른다.

- **문서화 규칙**:
  - 이 문서에서 고정한 파일 경로, 검증 기준, 후속 태스크 경계를 구현 단계와 동일하게 유지한다.
