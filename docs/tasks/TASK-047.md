## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 기보 목록과 상세 보기가 가능하지만, 잘못 기록된 기보를 제거할 수단이 없습니다.
- **이 작업의 책임**: 기보를 영구적으로 삭제하는 기능을 구현하며, 실수로 인한 삭제를 방지하기 위한 확인 절차를 포함합니다.

- **이번 작업에서 하지 않는 것**: 없음에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 기보 삭제 확인 및 처리 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🎯 1. 작업 목표

- **최종 상태**: 기보 삭제 확인 및 처리 동작이 요구사항 기준으로 고정되어야 한다.

- **이번 작업의 최소 결과물**:
  - `apps/web/src/features/game-delete/ui/delete-game-dialog.tsx`
  - `apps/web/src/features/game-delete/api/delete-game.ts`
  - `apps/web/src/pages/game-detail-page.tsx`
- **성공 기준 (AC)**:
  - 삭제 버튼 클릭 시 명시적인 확인 모달이 나타난다.
  - 삭제 완료 후 해당 기보가 목록에서 즉시 제거되고 페이지가 이동한다.
  - Soft Delete가 적용되어 실제 DB 레코드는 유지되면서 사용자에게만 안 보이게 처리됨을 확인했다.

## 🛠️ 3. 상세 기술 사양

- **UI**:
  - 기보 상세 화면의 헤더 또는 목록의 각 항목 메뉴에 삭제 버튼 배치.
  - `shadcn/ui`의 `AlertDialog` 사용: "이 기보를 영구적으로 삭제하시겠습니까? 삭제된 기보는 복구할 수 없습니다."
- **API 연동**:
  - `DELETE /api/games/:id` 호출.
  - 성공 시 `toast.success` 표시 후 목록 페이지로 리다이렉트.
- **백엔드 (API)**:
  - `GamesService.remove` 메서드 구현.
  - DB의 `deletedAt` 컬럼을 활용한 Soft Delete 권장 (프로젝트 규칙 준수).

- **핵심 조립/흐름 규칙**:
  - 삭제 버튼 클릭 시 명시적인 확인 모달이 나타난다.
  - 삭제 완료 후 해당 기보가 목록에서 즉시 제거되고 페이지가 이동한다.
  - Soft Delete가 적용되어 실제 DB 레코드는 유지되면서 사용자에게만 안 보이게 처리됨을 확인했다.

- **데이터 모델 해석**:
  - 입력은 삭제 대상 `gameId`와 현재 저장소/상세 컨텍스트입니다.
  - 출력은 삭제 mutation 상태와 목록/상세 화면 정리 결과입니다.

- **외부 의존성**:
  - `react`
  - `@tanstack/react-query`
  - `@/shared/ui/dialog`

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **권장 네이밍**:
  - 공개 함수/타입 이름: 기보 삭제 확인 및 처리 책임이 드러나는 이름
  - 내부 helper 이름: 역할이 바로 드러나는 동사형 또는 조합형 이름
  - 핵심 변수명: 상태와 대상이 분명한 이름
  - 피해야 할 이름: data, item, obj, temp

- **이름별 사용 의도와 적용 시점**:
  - `useDeleteGame`은 삭제 mutation과 후속 invalidate를 묶는 훅 이름으로 유지합니다.
  - `gameId`는 삭제 대상 기보 식별자 이름으로 사용합니다.

- **인수 이름 가이드**:
  - `gameId`, `repositoryId`, `onDeleted`

- **짧은 예시 골격**:

```ts
await deleteGame(gameId);
```

- **필수 describe/it 목록**:
  - 최상위 describe: `기보 삭제 확인 및 처리`
  - 필수 it: 요구사항 문서의 완료 기준을 직접 검증하는 테스트

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 상세 화면에서 삭제 성공 후 잘못된 경로에 남아 있는 경우
  - 삭제 확인 없이 바로 API를 호출하는 경우

## ⚖️ 4. 기술 제약 및 규칙

- **작성 원칙**:
  - 전역 규칙을 반복 나열하지 말고, 이번 태스크에서 특히 강조해야 하는 제약만 짧게 적는다.

- **구조 규칙**:
  - 현재 디렉토리 구조와 연관 설계 문서의 책임 경계를 유지한다.

  - 기존 상태를 직접 오염시키지 않고, 이번 태스크의 책임 범위 안에서 상태를 갱신한다.

- **범위 규칙**:
  - 없음에 연결된 범위는 여기서 닫지 않는다.

- **헌법 정렬 규칙**:
  - `../architecture/project-rules.md`의 네이밍, import/export, 상태 규칙을 그대로 따른다.

- **문서화 규칙**:
  - 이 문서에서 고정한 파일 경로, 검증 기준, 후속 태스크 경계를 구현 단계와 동일하게 유지한다.
