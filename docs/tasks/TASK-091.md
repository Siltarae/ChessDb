## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 로컬 커밋 훅(pre-commit)은 있으나, 원격 저장소 상에서 자동으로 코드 품질과 엄격한 테스트 커버리지를 보증해주는 시스템이 부재합니다.
- **이 작업의 책임**: CI 환경에서 빌드/린트/테스트를 자동화하고, 특히 핵심 로직에 대한 엄격한 테스트 커버리지 규칙을 시스템으로 강제합니다.

## 🎯 1. 작업 목표

- **최종 상태**: 코드를 푸시하거나 PR을 올리면 자동으로 품질 검사가 수행되며, 특히 `packages/shared`의 커버리지가 100% 미만이거나 전체 평균이 80% 미만일 경우 병합이 차단됩니다.
- **이번 작업의 결과물**:
  - `.github/workflows/ci.yml` (커버리지 옵션 포함)
- **성공 기준 (AC)**:
  - `pnpm turbo run test -- --coverage` 명령어가 CI 환경에서 실행되어야 한다.
  - 캐싱을 통해 CI 실행 속도가 최적화되어야 한다.
  - `packages/shared`에 대한 **100% 커버리지 규칙**이 명시적으로 설정되어야 한다.

## 📂 2. 대상 아티팩트

- **신규 생성**:
  - `.github/workflows/ci.yml`
- **수정 대상**:
  - 없음

## 🛠️ 3. 상세 기술 사양

- **CI 트리거**: `main`, `develop` 브랜치 Push/PR
- **엄격한 커버리지 게이트 (Gate)**:
  - `packages/shared`: **Line Coverage 100% 필수**
  - 전체 프로젝트 평균: **Line Coverage 80% 이상 필수**
- **최적화**: `actions/setup-node`의 `cache: 'pnpm'` 활용

## ⚖️ 4. 기술 제약 및 규칙

- **불변성**: CI 통과 없이는 `main` 브랜치 병합을 금지하는 것을 원칙으로 삼습니다. (GitHub 저장소 설정 권장)
- **무결성**: `packages/shared`는 체스 엔진의 핵심이므로 단 하나의 예외 케이스도 테스트 없이 허용하지 않습니다.

## 🧪 5. 검증 시나리오

1. **워크플로우 실행**: 파일 수정 후 푸시 시 Actions 탭에서 CI가 정상적으로 작동하는지 확인.
2. **커버리지 수집**: 테스트 실행 단계에서 커버리지 리포트가 생성되는지 로그 확인.

## 🚀 6. 권장 작업 순서

1. `.github/workflows/ci.yml` 작성 (커버리지 체크 로직 포함).

3. 코드 푸시 후 GitHub 웹사이트에서 CI 동작 모니터링.
