## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 로컬 개발자의 환경에서는 커밋 훅(pre-commit)이 보완을 해주나, 원격 저장소 상에서 자동으로 코드 품질(빌드/테스트)을 보증해주는 클라우드 CI는 부재한 상태입니다.
- **이 작업의 책임**: Main/Develop 브랜치 푸시 및 Pull Request 생성 시 작동하는 GitHub Actions 자동화 파이프라인을 구축합니다.
- **이번 작업에서 하지 않는 것**: 코드 배포(CD) 파이프라인은 작성하지 않습니다.
- **경계 메모**: 
  - 이후 작성되는 모든 앱과 패키지는 이 CI 파이프라인에서 `turbo` 명령어를 통해 자동으로 검사 대상에 포함됩니다.

## 🎯 1. 작업 목표

- **최종 상태**: GitHub 저장소에 코드가 반영되거나 병합 요청이 열릴 때마다 클라우드 러너에서 빌드, 린트, 테스트가 수행되며 PR 창에 성공/실패 여부를 노출합니다.
- **이번 작업의 최소 결과물**:
  - `.github/workflows/ci.yml`
- **성공 기준 (AC)**:
  - 저장소에 정의된 `pnpm`과 `Node.js` 버전을 러너가 동일하게 사용해야 한다.
  - 패키지 매니저(pnpm)의 캐싱을 활용해 워크플로우의 실행 속도를 최적화해야 한다.
  - 터미널 스크립트 `turbo run lint build test`를 에러 없이 실행해야 한다.

## 🛠️ 3. 상세 기술 사양

- **워크플로우 트리거 조건 (`on`)**:
  - `push` (대상 브랜치: `main`, `develop`)
  - `pull_request` (대상 브랜치: `main`, `develop`)
- **핵심 Steps 구조**:
  1. `actions/checkout@v4` : 코드베이스 확보
  2. `pnpm/action-setup@v3` : pnpm 글로벌 도구 세팅
  3. `actions/setup-node@v4` : Node 환경 세팅 (20.x, cache: 'pnpm')
  4. `pnpm install --frozen-lockfile` : 락파일 기반 의존성 정확한 동기화
  5. `pnpm turbo run lint build test` : 병렬 파이프라인 검증

## ⚖️ 4. 기술 제약 및 규칙

- **작성 원칙**: CI 비용 및 시간 절약을 위해 `pnpm cache` 설정을 반드시 적용합니다.
- **의존성 규칙**: 의존성 설치 시 예기치 못한 버전 변경을 막기 위해 `--frozen-lockfile` 플래그를 강제합니다.

## 🧪 5. 검증 시나리오 및 단언

1. **정상 시나리오: 워크플로우 트리거 및 통과**
   - 코드를 GitHub 원격 저장소에 푸시했을 때, Actions 탭에 새로운 실행 기록이 생성되며 성공(초록색 마크)으로 완료되어야 합니다.

## 🚀 6. 권장 작업 순서

1. 루트에 `.github/workflows` 폴더 경로를 생성합니다.
2. `ci.yml` 템플릿을 위 사양에 맞게 작성합니다.
3. 코드를 푸시하고 GitHub 웹페이지에서 동작을 모니터링합니다.
