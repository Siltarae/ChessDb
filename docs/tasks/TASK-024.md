## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 초안이 자동으로 복원되지만, 사용자가 새로운 대국을 기록하고 싶을 때 기존 내용을 비우는 기능이 없습니다.
- **이 작업의 책임**: 사용자가 '새 게임' 버튼을 눌렀을 때, 명시적인 확인 절차를 거쳐 기존 초안 데이터를 완전 삭제하고 초기 상태로 되돌립니다.

- **이번 작업에서 하지 않는 것**: `[TASK-025]`에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 기존 초안 폐기 후 새 기보 시작 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🎯 1. 작업 목표

- **최종 상태**: 사용자가 초기화 확인 모달에서 '수락'을 누르면 보드와 수순 목록이 깨끗이 비워지고 `localStorage`도 정리됩니다.

- **이번 작업의 최소 결과물**:
  - `apps/web/src/features/draft-management/ui/discard-draft-dialog.tsx`
  - `apps/web/src/features/draft-management/model/use-new-draft.ts`
  - `apps/web/src/entities/draft/model/draft-store.ts`
- **성공 기준 (AC)**:
  - 사용자 확인 모달이 정상적으로 작동하며 '취소' 시 데이터가 유지된다.
  - '확인' 클릭 후 메모리 상태와 로컬 스토리지 데이터가 동시에 제거된다.
  - 초기화 후 보드가 표준 시작 포지션으로 정확히 돌아온다.

## 📂 2. 대상 아티팩트

  - `[TASK-025]`에 연결된 후속 책임 파일

- **아티팩트 작성 규칙**:
  - 가능한 한 실제 파일 경로를 기준으로 작성하고, 범위 밖 파일은 이유 없이 함께 수정하지 않는다.
  - 수정 금지 범위나 후속 태스크 책임 파일은 이 섹션에서 명시적으로 분리한다.

## 🛠️ 3. 상세 기술 사양

- **UI**:
  - `shadcn/ui`의 `AlertDialog` 컴포넌트를 사용하여 "현재 입력 중인 내용이 모두 사라집니다. 계속하시겠습니까?" 메시지 표시.
- **로직 흐름**:
  1. `resetGame` 액션 호출 (Zustand).
  2. `localStorage.removeItem(LOCAL_STORAGE_DRAFT_KEY)` 실행.
  3. 초기 FEN 상태로 보드 재설정.
- **권장 네이밍**:
  - `ResetDraftDialog`: 초기화 확인 모달 컴포넌트.
  - `clearDraft`: 상태 및 저장소를 모두 비우는 액션 함수.

- **핵심 조립/흐름 규칙**:
  - 사용자 확인 모달이 정상적으로 작동하며 '취소' 시 데이터가 유지된다.
  - '확인' 클릭 후 메모리 상태와 로컬 스토리지 데이터가 동시에 제거된다.
  - 초기화 후 보드가 표준 시작 포지션으로 정확히 돌아온다.

- **데이터 모델 해석**:
  - 입력은 현재 draft 존재 여부와 폐기 확인 결과입니다.
  - 출력은 새 draft 초기화 또는 기존 draft 유지 분기입니다.

- **외부 의존성**:
  - `react`
  - `zustand`
  - `@chess-db/shared` 초기 상태 생성 로직

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **이름별 사용 의도와 적용 시점**:
  - `useNewDraft`는 새 기보 시작과 기존 초안 폐기 확인을 묶는 훅 이름으로 사용합니다.
  - `discardDraftDialogOpen`은 확인 다이얼로그 상태 이름으로 유지합니다.

- **인수 이름 가이드**:
  - `hasExistingDraft`, `confirmDiscard`

- **짧은 예시 골격**:

```ts
const { startNewDraft } = useNewDraft();
```

- **필수 describe/it 목록**:
  - 최상위 describe: `기존 초안 폐기 후 새 기보 시작`
  - 필수 it: 요구사항 문서의 완료 기준을 직접 검증하는 테스트

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 기존 초안이 있는데 확인 없이 즉시 초기화하는 경우
  - 초안 폐기 후 초기 상태를 제대로 재생성하지 못하는 경우

## ⚖️ 4. 기술 제약 및 규칙

- **작성 원칙**:
  - 전역 규칙을 반복 나열하지 말고, 이번 태스크에서 특히 강조해야 하는 제약만 짧게 적는다.

- **구조 규칙**:
  - 현재 디렉토리 구조와 연관 설계 문서의 책임 경계를 유지한다.

- **불변성/상태 규칙**:
  - 기존 상태를 직접 오염시키지 않고, 이번 태스크의 책임 범위 안에서 상태를 갱신한다.

- **범위 규칙**:
  - `[TASK-025]`에 연결된 범위는 여기서 닫지 않는다.

- **헌법 정렬 규칙**:
  - `../architecture/project-rules.md`의 네이밍, import/export, 상태 규칙을 그대로 따른다.

- **문서화 규칙**:
  - 이 문서에서 고정한 파일 경로, 검증 기준, 후속 태스크 경계를 구현 단계와 동일하게 유지한다.

## 🧪 5. 검증 시나리오 및 단언

1. **정상 시나리오: 핵심 동작 확인**
   - 사용자 확인 모달이 정상적으로 작동하며 '취소' 시 데이터가 유지된다.
   - 요구사항 문서의 완료 기준이 코드 또는 테스트에서 직접 확인되어야 한다.

2. **실패 시나리오: 범위 침범 차단**
   - 기존 초안이 있으면 확인 절차가 먼저 와야 한다.
   - 폐기 확정 뒤에는 새 draft 상태가 초기 포지션으로 재설정되어야 한다.

- **검증 시나리오 작성 규칙**:
  - 정상 흐름과 반려 흐름을 함께 적고, 가능하면 현재 테스트 파일 또는 후속 테스트 포인트와 연결한다.
