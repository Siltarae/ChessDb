## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 프로젝트 루트 폴더에 `docs/`, `AGENTS.md`, `.git/` 등 문서와 환경 설정 파일들만 존재하며, Node.js 프로젝트로서의 정체성(`package.json`)이나 모노레포 구조(`pnpm-workspace.yaml`)는 전혀 잡혀 있지 않은 백지상태입니다.
- **이 작업의 책임**: 프로젝트 전체의 루트 환경을 `pnpm` 모노레포 워크스페이스로 정의하고, 의존성 관리 및 엔진 버전을 고정하여 개발 환경의 일관성을 확보합니다.
- **이번 작업에서 하지 않는 것**: 하위 앱(`apps/`)이나 패키지(`packages/`)를 생성하거나, `Turborepo`, `TypeScript` 등의 개별 도구를 설치하는 작업은 후속 태스크(`TASK-065`, `TASK-066`)로 넘깁니다.
- **경계 메모**:
  - 이 태스크는 오직 '모노레포의 그릇'을 만드는 것에만 집중합니다.
  - 루트 `package.json`에 비즈니스 로직과 관련된 의존성을 추가하지 않도록 주의합니다.

## 🎯 1. 작업 목표

- **최종 상태**: 루트 디렉토리에서 `pnpm install`을 실행할 수 있고, 모노레포로서 하위 워크스페이스를 인식할 준비가 완료된 상태입니다.
- **이번 작업의 최소 결과물**:
  - `package.json` (루트)
  - `pnpm-workspace.yaml`
  - `.npmrc`
- **성공 기준 (AC)**:
  - 루트 `package.json`에 `private: true`와 `packageManager` 설정이 포함되어야 한다.
  - `pnpm-workspace.yaml`에서 `apps/*`와 `packages/*` 경로가 정의되어야 한다.
  - `.npmrc`에서 의존성 호이스팅 정책이 명시되어야 한다.

## 📂 2. 대상 아티팩트

- **신규 생성**:
  - `package.json`: 루트 프로젝트 메타데이터 및 엔진 요구사항 정의
  - `pnpm-workspace.yaml`: 워크스페이스 패키지 탐색 범위 정의
  - `.npmrc`: pnpm의 동작 방식(호이스팅 등) 제어
- **수정 대상**:
  - 없음
- **이번 작업에서 수정하지 않음**:
  - `docs/` 내의 모든 설계 및 요구사항 문서 (읽기 전용)
- **아티팩트 작성 규칙**:
  - 모든 파일은 프로젝트 루트(`.`)에 생성합니다.

- **조건부 정리 대상**: 필요할 때만 작성
  - placeholder, 임시 스켈레톤, 중복 export, 오래된 경로 표기

## 🛠️ 3. 상세 기술 사양

- **공개 함수/타입 시그니처**: 해당 없음 (인프라 설정)
- **핵심 조립/흐름 규칙**:
  - pnpm 워크스페이스는 파일 시스템의 `apps/`와 `packages/` 폴더를 독립된 패키지 단위로 관리합니다.
- **데이터 모델 해석**: 해당 없음
- **외부 의존성**:
  - Node.js >= 20.0.0
  - pnpm >= 9.0.0
- **import/export 규칙**: 해당 없음
- **권장 네이밍**:
  - 프로젝트 이름: `chess-db`
- **이름별 사용 의도와 적용 시점**: 해당 없음
- **인수 이름 가이드**: 해당 없음
- **짧은 예시 골격**:
  ```yaml
  # pnpm-workspace.yaml 예시
  packages:
    - 'apps/*'
    - 'packages/*'
  ```
- **필수 describe/it 목록**: 해당 없음 (인프라 설정은 쉘 명령어로 검증)
- **최소 테스트 개수**: 해당 없음
- **반드시 포함할 실패 시나리오**:
  - `package.json`에 `private: true` 누락 시 루트 패키지가 실수로 배포될 위험이 있음.
  - `packageManager` 설정 누락 시 다른 패키지 매니저(npm, yarn) 사용으로 인한 잠금 파일 충돌 위험이 있음.

## ⚖️ 4. 기술 제약 및 규칙

- **작성 원칙**: `pnpm`의 엄격한 의존성 관리 원칙을 준수합니다.
- **구조 규칙**: 모든 하위 프로젝트는 반드시 `apps/` 또는 `packages/` 폴더 내부에 위치해야 합니다.
- **불변성/상태 규칙**: 해당 없음
- **범위 규칙**: 루트 `package.json`에는 전역 개발 도구(`turbo`, `eslint` 등) 외의 라이브러리는 설치하지 않습니다.
- **헌법 정렬 규칙**: `package.json` 내의 필드 순서는 보편적인 컨벤션을 따릅니다.
- **문서화 규칙**: 없음

## 🧪 5. 검증 시나리오 및 단언

1. **정상 시나리오: pnpm 설치 및 워크스페이스 확인**
   - `pnpm install` 실행
   - `node_modules`와 `pnpm-lock.yaml`이 생성되어야 함.
   - `pnpm m ls` (또는 `pnpm list -r`) 실행 시 루트 프로젝트가 표시되어야 함.

2. **실패 시나리오: 엔진 버전 불일치**
   - Node.js 버전을 20 미만으로 변경 후 `pnpm install` 실행
   - `engines` 설정에 의해 설치가 거부되거나 경고가 발생해야 함.

3. **경계 시나리오: 워크스페이스 경로 인식**
   - 빈 폴더 `apps/dummy` 생성 후 내부에 `package.json` 작성
   - `pnpm m ls` 실행 시 `apps/dummy`가 워크스페이스 구성원으로 나타나야 함.

- **검증 시나리오 작성 규칙**:
  - 정상 흐름과 반려 흐름을 함께 적고, 가능하면 현재 테스트 파일 또는 후속 테스트 포인트와 연결한다.

## 🚀 6. 권장 작업 순서

1. **문맥 확인**: `docs/architecture/tech-stack.md`에서 pnpm 및 모노레포 결정을 다시 확인합니다.
2. **패키지 초기화**: 루트에서 `pnpm init`을 실행하여 기본 `package.json`을 생성합니다.
3. **메타데이터 수정**: `package.json`을 열어 `private`, `engines`, `packageManager` 필드를 추가/수정합니다.
4. **워크스페이스 설정**: `pnpm-workspace.yaml` 파일을 생성하고 경로를 기입합니다.
5. **동작 설정**: `.npmrc` 파일을 생성하고 호이스팅 관련 설정을 추가합니다.
6. **검증 실행**: `pnpm install`을 실행하여 설정이 유효한지 확인합니다.
7. **자가 점검**: `apps/`와 `packages/` 폴더가 실제로 존재하는지 확인합니다(없다면 빈 디렉토리로 생성).
