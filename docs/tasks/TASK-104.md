## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 저장소 기본 보드뷰에는 저장 버튼 위치를 둘 수 있지만, 기존 기보를 업데이트하는 저장 계약은 아직 없습니다.
- **이 작업의 책임**: 기존 기보를 연 상태에서 저장 버튼을 누르면 새 기보 생성이 아니라 기존 `gameId`를 업데이트하도록 연결합니다.
- **이번 작업에서 하지 않는 것**: 저장소 기본 보드뷰의 최초 화면 구성은 `[TASK-034]`에서 다루며, 저장소 목록 조회는 `[TASK-037]`에서 다룹니다.
- **경계 메모**:
  - 새 기보 생성(`POST /api/games`)과 기존 기보 업데이트(`PATCH /api/games/:gameId`)를 명확히 분리합니다.

## 🎯 1. 작업 목표

- **최종 상태**: 사용자가 저장소 보드에서 기존 기보를 수정한 뒤 저장하면 해당 기보가 업데이트됩니다.
- **이번 작업의 최소 결과물**:
  - `apps/api/src/modules/games/**`
  - `apps/web/src/features/save-game/**` 또는 신규 update feature
  - `apps/web/src/pages/repository-home-page.tsx`
- **성공 기준 (AC)**:
  - 기존 기보 저장은 `gameId` 기반 업데이트 요청으로 실행된다.
  - 새 기보 저장과 기존 기보 업데이트 저장이 테스트에서 구분된다.
  - 저장 후 화면 상태가 업데이트된 서버 상태와 어긋나지 않는다.
  - 저장소 컨텍스트의 `repositoryId`와 업데이트 대상 `gameId`가 함께 유지된다.

## 📂 2. 대상 아티팩트

- **신규 생성**:
  - 필요 시 `apps/web/src/features/game-update-save/**`
- **수정 대상**:
  - `apps/api/src/modules/games/**`
  - `apps/web/src/pages/repository-home-page.tsx`
  - `apps/web/src/features/save-game/**`
- **이번 작업에서 수정하지 않음**:
  - 저장소 목록 UI
  - 저장소 삭제 UI
  - 통계/후보수 분석 UI

## 🛠️ 3. 상세 기술 사양

- 저장 대상이 새 기보인지 기존 기보인지 page 상태에서 구분합니다.
- 기존 기보 업데이트는 `gameId`를 필수 입력으로 받습니다.
- 저장소 범위 보장을 위해 `repositoryId`도 함께 유지합니다.
- API는 기존 기보 업데이트 전용 endpoint를 사용합니다.
- 수순 수정으로 인해 이후 수순이 제거되는 경우 코멘트와 어노테이션 정리 규칙을 함께 적용합니다.

## ⚖️ 4. 기술 제약 및 규칙

- 기존 기보 업데이트를 새 기보 생성 API로 처리하지 않습니다.
- `boardId` 같은 별도 저장 단위는 만들지 않고, 저장 대상은 `Game`으로 유지합니다.
- 불법 수나 서로 맞지 않는 SAN/보드 상태를 저장하지 않습니다.

## 🧪 5. 검증 시나리오 및 단언

1. **정상 시나리오: 기존 기보 업데이트 저장**
   - 기존 `gameId`가 있는 상태에서 저장
   - `PATCH /api/games/:gameId` 요청이 실행된다.

2. **실패 시나리오: 새 기보로 잘못 저장**
   - 기존 `gameId`가 있는데 `POST /api/games`가 호출된다.
   - 테스트가 실패해야 한다.

3. **정상 시나리오: 저장소 컨텍스트 유지**
   - `repositoryId`와 `gameId`가 모두 있는 상태에서 저장
   - 다른 저장소의 기보로 업데이트되지 않는다.
