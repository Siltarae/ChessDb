## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 분석 보드와 기보 상세 화면이 각각 독립적으로 작동하고 있습니다. 특정 기보를 복기하다가 궁금한 시점부터 분석을 시작할 수 있는 연결 고리가 필요합니다.
- **이 작업의 책임**: 기보 상세 화면에서 특정 수순의 포지션을 분석 보드로 전송하고 페이지를 전환하는 기능을 구현합니다.

- **이번 작업에서 하지 않는 것**: `[TASK-054]` (통계 집계)에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 저장 기보 기반 포지션 진입 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🎯 1. 작업 목표

- **최종 상태**: 사용자가 과거 대국의 특정 수순에서 '분석 보드에서 열기'를 누르면, 해당 시점의 기물 배치 그대로 분석 도구가 열려 다양한 후보 수의 승률을 확인할 수 있습니다.

- **이번 작업의 최소 결과물**:
  - `apps/web/src/features/analysis-entry/model/start-analysis-from-game.ts`
  - `apps/web/src/pages/game-detail-page.tsx`
  - `apps/web/src/pages/analysis-board-page.tsx`
- **성공 기준 (AC)**:
  - 버튼 클릭 시 분석 보드 페이지로 즉시 전환된다.
  - 상세 뷰에서 보고 있던 보드 상태(기물 위치, 턴 등)가 분석 보드에 완벽히 복제된다.
  - 분석 보드 진입 후 해당 시점부터 새로운 수를 두어볼 수 있다.

## 🛠️ 3. 상세 기술 사양

- **데이터 연동**:
  - 상세 뷰에서 현재 보고 있는 `GameState`를 `AnalysisStore`의 `setInitialState` 액션으로 전달.
  - `react-router`의 `useNavigate`를 사용하여 분석 보드 URL로 이동.
- **UI**:
  - 기보 상세 페이지 하단 컨트롤 바에 '이 포지션 분석하기' 아이콘/버튼 추가.

- **핵심 조립/흐름 규칙**:
  - 버튼 클릭 시 분석 보드 페이지로 즉시 전환된다.
  - 상세 뷰에서 보고 있던 보드 상태(기물 위치, 턴 등)가 분석 보드에 완벽히 복제된다.
  - 분석 보드 진입 후 해당 시점부터 새로운 수를 두어볼 수 있다.

- **데이터 모델 해석**:
  - 입력은 저장된 기보의 특정 `gameId` 또는 초기 분석 진입에 필요한 포지션 정보입니다.
  - 출력은 분석 보드가 소비할 초기 포지션 상태와 이동 경로입니다.

- **외부 의존성**:
  - `react-router-dom`
  - `@tanstack/react-query`
  - `@chess-db/shared` `GameState`

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **권장 네이밍**:
  - 공개 함수/타입 이름: 저장 기보 기반 포지션 진입 책임이 드러나는 이름
  - 내부 helper 이름: 역할이 바로 드러나는 동사형 또는 조합형 이름
  - 핵심 변수명: 상태와 대상이 분명한 이름
  - 피해야 할 이름: data, item, obj, temp

- **이름별 사용 의도와 적용 시점**:
  - `startAnalysisFromGame`은 저장 기보를 분석 보드 진입 상태로 변환할 때 사용합니다.
  - `sourceGameId`는 원본 기보 식별자 이름으로 유지합니다.

- **인수 이름 가이드**:
  - `gameId`, `sourceGameId`, `initialAnalysisState`

- **짧은 예시 골격**:

```ts
const initialAnalysisState = await startAnalysisFromGame(gameId);
```

- **필수 describe/it 목록**:
  - 최상위 describe: `저장 기보 기반 포지션 진입`
  - 필수 it: 요구사항 문서의 완료 기준을 직접 검증하는 테스트

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 저장 기보의 마지막 상태와 다른 포지션으로 분석 보드를 여는 경우
  - 원본 기보 식별 없이 임시 상태만 전달해 재현성이 떨어지는 경우
