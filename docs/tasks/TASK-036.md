## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: 저장소 목록 조회가 가능하지만, 잘못 만들었거나 더 이상 필요 없는 저장소를 제거할 방법이 없습니다.
- **이 작업의 책임**: 저장소를 삭제할 때 발생할 수 있는 데이터 손실(하위 기보 모두 삭제)을 사용자에게 알리고, 안전하게 삭제 요청을 처리합니다.

- **이번 작업에서 하지 않는 것**: 없음에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 저장소 삭제 확인 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🎯 1. 작업 목표

- **최종 상태**: 삭제 버튼 클릭 시 강력한 경고 메시지가 노출되며, 확인 후 서버에서 데이터가 영구적으로 제거됩니다.

- **이번 작업의 최소 결과물**:
  - `apps/web/src/features/repository-delete/ui/delete-repository-dialog.tsx`
  - `apps/web/src/features/repository-delete/api/delete-repository.ts`
  - `apps/web/src/pages/repository-list-page.tsx`
- **성공 기준 (AC)**:
  - 삭제 전 반드시 사용자의 명시적인 확인(Confirm) 절차를 거친다.
  - 삭제 성공 후 목록에서 해당 데이터가 즉시 사라짐을 확인했다.
  - 삭제된 저장소의 URL로 직접 접근 시 404 에러 또는 목록으로 리다이렉트 처리된다.

## 📂 2. 대상 아티팩트

- **신규 생성**:
  - `apps/web/src/features/repository-delete/ui/delete-repository-dialog.tsx`
  - `apps/web/src/features/repository-delete/api/delete-repository.ts`
- **수정 대상**:
  - `apps/web/src/pages/repository-list-page.tsx`
- **조건부 정리 대상**: 필요할 때만 작성
  - placeholder, 임시 스켈레톤, 중복 export, 오래된 경로 표기

- **이번 작업에서 수정하지 않음**:
  - 없음에 연결된 후속 책임 파일

- **아티팩트 작성 규칙**:
  - 가능한 한 실제 파일 경로를 기준으로 작성하고, 범위 밖 파일은 이유 없이 함께 수정하지 않는다.
  - 수정 금지 범위나 후속 태스크 책임 파일은 이 섹션에서 명시적으로 분리한다.

## 🛠️ 3. 상세 기술 사양

- **UI**:
  - 저장소 카드 내 또는 상세 페이지 관리 메뉴에 삭제 버튼 배치.
  - `shadcn/ui`의 `AlertDialog` 사용: "정말로 삭제하시겠습니까? 이 작업은 되돌릴 수 없으며 포함된 모든 기보가 사라집니다."
- **API 연동**:
  - `DELETE /api/repositories/:repoId` 호출.
  - 성공 시 목록 페이지로 이동하거나 목록을 갱신.
- **백엔드 (API)**:
  - `Prisma`의 `onDelete: Cascade` 설정을 확인하거나 서비스 단에서 연관된 기보들을 먼저 처리.

- **핵심 조립/흐름 규칙**:
  - 삭제 전 반드시 사용자의 명시적인 확인(Confirm) 절차를 거친다.
  - 삭제 성공 후 목록에서 해당 데이터가 즉시 사라짐을 확인했다.
  - 삭제된 저장소의 URL로 직접 접근 시 404 에러 또는 목록으로 리다이렉트 처리된다.

- **데이터 모델 해석**:
  - 입력은 삭제 대상 `repositoryId`와 사용자 확인 결과입니다.
  - 출력은 삭제 mutation 성공/실패 상태와 목록 invalidate 결과입니다.

- **외부 의존성**:
  - `react`
  - `@tanstack/react-query`
  - `@/shared/ui/dialog`

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **권장 네이밍**:
  - 공개 함수/타입 이름: 저장소 삭제 확인 책임이 드러나는 이름
  - 내부 helper 이름: 역할이 바로 드러나는 동사형 또는 조합형 이름
  - 핵심 변수명: 상태와 대상이 분명한 이름
  - 피해야 할 이름: data, item, obj, temp

- **이름별 사용 의도와 적용 시점**:
  - `useDeleteRepository`는 삭제 mutation과 확인 흐름을 함께 다룰 때 사용합니다.
  - `repositoryId`는 삭제 대상 식별자 이름으로 유지합니다.

- **인수 이름 가이드**:
  - `repositoryId`, `onConfirmDelete`

- **짧은 예시 골격**:

```ts
await deleteRepository(repositoryId);
```

- **필수 describe/it 목록**:
  - 최상위 describe: `저장소 삭제 확인`
  - 필수 it: 요구사항 문서의 완료 기준을 직접 검증하는 테스트

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 삭제 확인 없이 즉시 mutation을 호출하는 경우
  - 삭제 성공 후 목록이 갱신되지 않는 경우
