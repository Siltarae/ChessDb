## 0. 현재 코드 상태와 이 작업의 위치

- **현재 상태 요약**: NestJS 초기화 시 생성된 기본 테스트 코드는 있으나, 모노레포 구조와 공유 패키지를 고려한 정밀한 테스트 설정은 부재합니다.
- **이 작업의 책임**: 서비스 로직을 검증하는 단위 테스트(Jest)와 실제 API 호출을 시뮬레이션하는 E2E 테스트 환경을 최적화합니다.

- **이번 작업에서 하지 않는 것**: `[TASK-094]` (환경 변수 검증)에 연결된 후속 책임은 이번 태스크에서 함께 닫지 않는다.

- **경계 메모**:
  - 이번 태스크는 apps/api Jest 및 Supertest 기반 테스트 환경 구성 범위만 닫고, 후속 태스크 또는 인접 Feature의 세부 구현은 여기서 함께 처리하지 않는다.

## 🛠️ 3. 상세 기술 사양

- **도구**: `jest`, `ts-jest`, `supertest`.
- **단위 테스트**: 각 서비스 및 컨트롤러 파일 옆에 `.spec.ts` 배치.
- **E2E 테스트**: `apps/api/test/` 폴더 내에서 전체 HTTP 흐름 검증.
- **모노레포 연동**: `jest.config.ts`에서 `@chess-db/shared` 및 기타 별칭 경로를 `moduleNameMapper`에 정확히 등록.
- **필수 describe/it 목록**:
  - describe: `API Integration Testing`
    - it: `E2E 테스트 환경에서 DB Mocking 또는 Test DB 연결이 정상 작동해야 한다`
    - it: `모노레포 공유 패키지의 코드를 Jest 환경에서 인식해야 한다`

- **핵심 조립/흐름 규칙**:
  - 단위 테스트와 E2E 테스트 명령어가 각각 정상적으로 수행된다.
  - 테스트 환경에서도 환경 변수(`.env.test`)가 올바르게 주입된다.
  - `shared` 패키지 코드를 사용하는 서비스에 대한 테스트가 성공한다.

- **데이터 모델 해석**:
  - 핵심 데이터는 Jest 설정, e2e용 Supertest 진입점, Nest 테스트 부트스트랩입니다.
  - 입력은 API 앱 테스트 범위와 Node 환경 설정 파일입니다.

- **외부 의존성**:
  - `jest`
  - `supertest`
  - `@nestjs/testing`

- **import/export 규칙**:
  - `../architecture/project-rules.md`의 named export, 상대 경로 최소화, `.js` 확장자 규칙을 따른다.

- **권장 네이밍**:
  - 공개 함수/타입 이름: apps/api Jest 및 Supertest 기반 테스트 환경 구성 책임이 드러나는 이름
  - 내부 helper 이름: 역할이 바로 드러나는 동사형 또는 조합형 이름
  - 핵심 변수명: 상태와 대상이 분명한 이름
  - 피해야 할 이름: data, item, obj, temp

- **이름별 사용 의도와 적용 시점**:
  - `app.e2e-spec.ts`, `jest.config.ts`는 API 테스트 진입 계약 파일 이름으로 유지합니다.
  - `createTestingModule`은 Nest 테스트 부트스트랩을 표현할 때 사용합니다.

- **인수 이름 가이드**:
  - `app`, `moduleRef`, `request`

- **짧은 예시 골격**:

```ts
const moduleRef = await Test.createTestingModule({ imports: [AppModule] }).compile();
```

- **최소 테스트 개수**:
  - 최소 3개

- **반드시 포함할 실패 시나리오**:
  - 유닛 테스트와 e2e 테스트 설정을 섞어 Supertest 실행이 안 되는 경우
  - 테스트 부트스트랩에서 실제 AppModule과 다른 모듈을 여는 경우
