# EPIC-001 학습용 기보 저장 및 분석

## 개요

이 Epic은 작성자 본인이 주 사용자로 사용하는 체스 학습용 웹사이트를 정의한다.
서비스는 개인 사용 목적이 우선이며, 함께 공개 배포할 수 있는 형태를 지향한다.
사용자는 기보를 직접 입력하고, 저장하고, 다시 보며, 분석에 활용할 수 있어야 한다.

## 목표

- 사용자가 기보를 직접 입력할 수 있어야 한다.
- 사용자가 자신의 기보를 저장할 수 있어야 한다.
- 저장한 기보를 다시 조회할 수 있어야 한다.
- 저장한 기보를 학습 목적에 맞게 분석할 수 있어야 한다.

## 범위 초안

- 체스보드 기반 기보 입력
- 기보 저장
- 기보 조회
- 기보 분석

## 현재 합의

- 주 사용자는 작성자 본인이다.
- 개인 사용 목적이 우선이고, 공개 배포는 부가 목적이다.
- 초기 핵심 기능은 기보 저장 자체보다 기보 입력이다.
- 기보 입력은 체스보드에서 수를 직접 두는 방식이 우선이다.
- 초기 입력 기능은 합법 수만 허용해야 한다.
- 사용자가 기물을 클릭했을 때 해당 기물의 합법 수를 보드에 표시할 수 있어야 한다.
- 초기 버전은 새 게임을 처음부터 입력하는 흐름을 우선한다.
- 사용자가 둔 수는 수순 목록으로 함께 보여야 한다.
- 각 수에 대해 학습 메모 성격의 코멘트를 남길 수 있어야 한다.
- 코멘트는 각 반수마다 하나씩 작성할 수 있으면 충분하다.
- 각 수에는 착수 평가 기호 형태의 어노테이션을 붙일 수 있어야 한다.
- 사용자는 입력한 수에 대해 되돌리기와 다시하기를 할 수 있어야 한다.
- 되돌리기 이후 새 수를 두면 기존 이후 수열은 삭제되는 방식이면 충분하다.
- 사용자가 입력 중이던 기보는 브라우저 새로고침 이후에도 유지되어야 한다.
- 유지 범위에는 보드 상태, 수순 목록, 반수별 코멘트, 착수 평가 기호가 모두 포함되어야 한다.
- 사용자는 여러 저장소를 생성할 수 있어야 한다.
- 사용자는 저장소 이름을 직접 정할 수 있어야 한다.
- 사용자는 여러 저장소 안에 여러 기보를 저장하고 다시 불러올 수 있어야 한다.
- 사용자는 저장소를 삭제할 수 있어야 한다.
- 저장소 삭제는 사용자 확인 이후에만 수행되어야 한다.
- 초기 저장소 목록은 생성한 순서대로 보여주면 충분하다.
- 사용자는 저장소를 먼저 선택해 진입한 뒤, 그 저장소 안에서 기보를 작성하고 정식 저장하는 흐름으로 사용할 수 있어야 한다.
- 저장소 안에서는 사용자가 새 기보 작성을 명시적으로 시작할 수 있어야 한다.
- 기존 초안이 남아 있을 때 새 기보 작성을 시작하면, 사용자 확인 후 기존 초안을 버리고 새로 시작해야 한다.
- 사용자는 저장한 기보를 삭제할 수 있어야 한다.
- 기보 삭제는 사용자 확인 이후에만 수행되어야 한다.
- 초기 기보 목록은 날짜순으로 나열할 수 있으면 충분하다.
- 기보 저장은 자동 저장과 수동 저장을 모두 지원해야 한다.
- 자동 저장은 수를 둘 때마다 바로 반영되는 방식이면 충분하다.
- 수동 저장은 자동 저장 중인 초안을 사용자가 정식 기보로 확정하는 용도여야 한다.
- 정식 저장 전 초안은 항상 1개만 유지되면 충분하다.
- 정식 저장된 기보의 수순은 수정하지 않는다.
- 정식 저장된 기보에도 반수별 코멘트와 착수 평가 기호는 추가하거나 수정할 수 있어야 한다.
- 정식 저장된 기보에서 코멘트와 착수 평가 기호를 수정하면 즉시 자동 저장되어야 한다.
- 저장한 여러 기보를 기반으로 학습용 통계를 낼 수 있어야 한다.
- 초기 통계는 특정 수순이나 포지션 다음에 많이 둔 수와 해당 수의 승률을 보여줄 수 있어야 한다.
- 통계의 기본 집계 단위는 저장소여야 한다.
- 각 기보에는 게임 결과를 사용자가 직접 기록할 수 있어야 한다.
- 결과는 체크메이트만이 아니라 시간승/패처럼 외부 요인으로 끝난 경우도 표현할 수 있어야 한다.
- 각 기보에는 게임 결과와 종료 사유를 분리해서 기록할 수 있어야 한다.
- 초기 종료 사유는 체크메이트, 시간패, 기권, 무승부 4가지면 충분하다.
- 각 기보에는 대국 날짜를 함께 기록할 수 있어야 한다.
- 승률 표시는 백승, 무승부, 흑승 기준으로 제공할 수 있어야 한다.

## 메모

- 현재는 Epic 요구사항을 먼저 충분히 구체화하는 단계다.
- 상대 이름은 현재 범위에서 필수 요구사항으로 보지 않는다.
- 오프닝이나 특정 수순 기준의 고급 필터링 통계는 현재 범위에서 제외한다.
- 초기 통계 화면은 특정 포지션에서 다음 수 후보와 승률을 조회하는 기능이면 충분하다.
- 통계용 포지션 진입 방식은 저장된 기보를 따라 들어가는 방식과 별도 입력 방식이 모두 필요하다.
- 별도 입력 방식은 보드에서 수를 직접 두어 포지션을 만드는 방식이면 충분하다.
- 초기 버전은 로그인 없이 사용하는 단일 사용자 환경을 전제로 한다.
- 저장소를 여러 개 두는 방식으로 사용자가 스스로 학습 맥락을 나눌 수 있어야 한다.
- 백/흑 분리는 시스템 기본 축으로 강제하지 않고, 필요하면 사용자가 저장소를 나눠 관리하는 방식으로 해결한다.
- 저장소 복사와 저장소 간 붙여넣기는 후순위 범위로 둔다.
- 저장소 내부 UI는 한 화면에 모든 기능을 동시에 노출하기보다, 역할에 따라 뷰를 나누는 방향이 적합하다.
- 저장소 내부에서는 기보 목록 중심 화면, 분석 보드 뷰, 기보 입력 뷰, 개별 기보 상세 화면을 분리하는 방향을 우선 검토한다.
- 분석 보드 뷰는 새 기보 입력 전용 화면이 아니라, 보드를 기준으로 포지션을 만들고 통계를 조회하는 작업 화면을 뜻한다.
- 저장소에 진입했을 때는 분석 보드 뷰가 가장 먼저 보여야 한다.
- 분석 보드 뷰에서 새 기보 작성 버튼을 누르면 기보 입력 뷰로 자연스럽게 전환되어야 한다.
- 기보 상세 뷰는 저장소 내 기보 목록에서 특정 기보를 선택했을 때 들어가는 화면이어야 한다.
- 기보 입력 뷰에서 정식 저장을 마친 뒤에는 해당 기보의 상세 뷰로 이동하는 흐름이 자연스럽다.
- 기보 상세 뷰에서는 보드, 수순 목록, 코멘트와 착수 평가 기호를 함께 볼 수 있어야 한다.
- 분석 보드 뷰는 통계 조회 중심 화면으로 이해할 수 있어야 한다.
- 기보 상세 뷰는 저장된 기보의 코멘트와 어노테이션 확인 및 보완 중심 화면으로 이해할 수 있어야 한다.
- 기보 상세 뷰에서는 사용자가 수순 목록의 특정 수를 선택하면 해당 시점의 보드 상태로 이동할 수 있어야 한다.
- 분석 보드 뷰에서도 포지션 입력을 위해 되돌리기와 다시하기를 지원하는 편이 좋다.
- Variation 보존은 현재 범위에서 제외하고 후순위 확장으로 둔다.
