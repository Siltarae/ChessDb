# 핵심 패턴

## 목적

- 프로젝트 전반에 적용할 설계 패턴과 코드 구조 원칙을 기록한다.

## 작성 원칙

- 반복해서 적용되는 패턴만 기록한다.
- 특정 Task에만 해당하는 구현 방식은 Task 문서에 남긴다.
- 패턴 이름만 적지 말고, 왜 쓰는지 합의가 생기면 함께 적는다.

## 상태 관리 패턴

- **서버 상태와 클라이언트 상태 분리**
  - **적용 이유**: API를 통해 가져온 비동기 데이터(기보 목록 등)는 `TanStack Query`가 전담하고, 순수 UI/게임 상태(보드 상태, 현재 턴 등)는 `Zustand`가 전담하여 상태의 성격과 생명주기를 명확히 분리하기 위함.

## 데이터 흐름 패턴

- **REST 기반 단방향 통신**
  - **적용 이유**: 프론트엔드는 백엔드에 REST API 요청을 보내고 백엔드는 검증 후 데이터를 반환하는 표준적인 단방향 흐름을 강제하여 데이터 추적성을 높임.

## 모듈 구성 패턴

- **프론트엔드: Feature-Sliced Design (FSD)**
  - **적용 이유**: 체스 보드, 기보 목록, 통계 등 복잡한 도메인 요구사항을 계층(`app`, `pages`, `widgets`, `features`, `entities`, `shared`)별로 엄격하게 분리하여 UI와 비즈니스 로직의 결합도를 낮추고 유지보수성을 극대화하기 위함.
- **백엔드: Pragmatic 모듈 분리 (Controller - Service - Repository)**
  - **적용 이유**: NestJS의 직관적인 폴더 구조(빠른 개발)를 유지하면서도, Service(비즈니스 로직)와 Repository(DB 통신)를 철저히 분리하여 로직의 순수성과 테스트 용이성을 확보하기 위함.
