# 통합 규칙

## 목적

- 모듈 간 연결 방식과 의존 규칙을 기록한다.

## 작성 원칙

- 반복 적용되는 연결 규칙만 기록한다.
- import 방향, 의존성 경계, 인터페이스 연결 규칙을 우선 정리한다.
- 특정 Task에만 해당하는 예외는 Task 문서에 남긴다.

## 핵심 데이터 흐름 및 로직 분리 방안

- **FEN 및 SAN 파싱 주체: 양방향 공유 검증 (Dual Validation)**
  - 프론트엔드가 `packages/shared` 체스 엔진을 활용해 다음 FEN과 SAN을 즉시 계산하여 UI를 낙관적으로 갱신(Optimistic Update)한다.
  - 계산된 결과를 백엔드로 전송하면, 백엔드 역시 동일한 `packages/shared` 엔진으로 서버 사이드 검증을 한 번 더 수행한 뒤 데이터베이스에 최종 저장한다.
  - 이 방식을 통해 **프론트엔드의 빠른 반응성**과 **백엔드의 견고한 데이터 무결성**을 동시에 확보한다.

## 의존 방향 규칙

- **모노레포 패키지 간 단방향 의존**
  - `apps/frontend` ➡️ `packages/shared` (허용)
  - `apps/backend` ➡️ `packages/shared` (허용)
  - `packages/shared` ➡️ `apps/*` (절대 금지: shared는 특정 앱에 의존하지 않는 순수 로직 유지)
  - `apps/frontend` ↔️ `apps/backend` (직접 참조 절대 금지: 오직 REST API로만 통신)

- **프론트엔드 FSD 계층 간 단방향 의존**
  - 상위 계층은 하위 계층을 참조할 수 있으나, 하위 계층은 상위 계층을 참조할 수 없다.
  - 참조 방향: `app` ➡️ `pages` ➡️ `widgets` ➡️ `features` ➡️ `entities` ➡️ `shared`

## 인터페이스 연결 규칙

- **프론트엔드-백엔드 데이터 연동**
  - 두 애플리케이션 간의 통신은 `packages/shared`에 정의된 Zod 스키마 및 DTO(타입)를 기준으로 수행하여 런타임/컴파일 타임의 타입 안정성을 모두 확보한다.

## 외부 시스템 연동 규칙

- 현재는 프론트엔드와 백엔드 간의 통신만 정의하며, 외부 API 연동 필요시 별도의 어댑터 계층을 둔다.

## 예외 처리 원칙

- **프론트엔드-백엔드 간 에러 규약**
  - 백엔드는 클라이언트가 이해할 수 있는 표준 HTTP 상태 코드와 명확한 에러 코드/메시지를 포함한 공통 JSON 포맷으로 에러를 반환한다.
