# 기술 스택 (Tech Stack)

## 목적

- 프로젝트에서 사용할 기술과 선택 이유를 기록한다.

## 작성 원칙

- 실제로 합의된 기술 선택만 기록한다.
- 후보 비교가 필요하면 선택 이유와 제외 이유를 함께 적는다.
- 아직 결정되지 않은 내용은 억지로 채우지 않는다.

## 프론트엔드

- **기반 프레임워크**: React + Vite (SPA)
  - **선택 이유**: 체스 보드 렌더링과 PGN 파싱 등 클라이언트 측 상태 관리가 중요하며, 가볍고 빠른 개발 환경을 구성하기 위함.
- **상태 관리**: Zustand
  - **선택 이유**: 보일러플레이트가 적고 직관적인 API를 제공하여 체스 보드 및 게임 상태를 가볍고 효율적으로 관리하기 위함.
- **UI 라이브러리 및 스타일링**: shadcn/ui + Tailwind CSS
  - **선택 이유**: 컴포넌트 코드를 직접 소유하여 커스터마이징 자유도가 높고, 유틸리티 클래스 기반으로 빠르고 일관된 UI 개발이 가능하기 때문.
- **필수 유틸리티 라이브러리**: dnd-kit (드래그 앤 드롭), TanStack Query (서버 상태 동기화), React Router (클라이언트 라우팅)
  - **선택 이유**: 체스 기물 조작, API 데이터 캐싱, 화면 전환 등 복잡한 프론트엔드 요구사항을 안정적으로 처리하기 위함.

## 체스 도메인 및 렌더링

- **구현 방식**: 외부 라이브러리 없이 전체 직접 구현 (Custom Implementation)
  - **선택 이유**: 체스 행마법 검증, PGN 파싱, 보드 렌더링 등 핵심 로직에 대한 완벽한 통제권을 확보하고, 프로젝트 요구사항에 맞는 커스텀 데이터 구조를 유연하게 설계하기 위함.
- **코어 로직 배치**: 공통 패키지 (`packages/shared`)
  - **선택 이유**: UI에 의존하지 않는 순수 체스 도메인 로직(규칙, 상태, PGN 파싱)을 프론트엔드와 백엔드 양쪽에서 재사용하고 데이터 일관성을 유지하기 위함.

## 백엔드

- **기반 프레임워크**: NestJS
  - **선택 이유**: 안정적인 아키텍처와 타입스크립트 기반의 강력한 생태계를 활용하여 확장성 있는 서버를 구축하기 위함.

## 데이터 저장소

- **데이터베이스 및 ORM**: PostgreSQL + Prisma
  - **선택 이유**: 체스 기보 데이터의 유연한 저장(JSONB 등)과 관계형 데이터의 무결성을 동시에 만족하며, NestJS 생태계와의 호환성 및 개발 생산성이 가장 뛰어나기 때문.

## API 통신 및 환경

- **통신 방식**: REST API + 공유 DTO
  - **선택 이유**: NestJS의 기본 생태계와 가장 잘 어울리며, 모노레포 환경에서 프론트엔드와 백엔드가 타입(DTO)을 공유하여 안정적이고 직관적인 통신을 구현하기 위함.
- **데이터 검증 및 환경 변수 관리**: Zod + `@nestjs/config`
  - **선택 이유**: 프론트엔드와 백엔드가 공통 패키지에서 스키마를 공유하여 런타임 데이터 검증을 수행하고, 앱(서버/클라이언트) 실행 시점에 필수 환경 변수(`.env`)의 유무와 타입을 강제로 검증(Fail-Fast)하기 위함.
- **API 문서화**: `@nestjs/swagger`
  - **선택 이유**: 백엔드의 REST API 명세를 코드로 자동화하여 프론트엔드 개발 시 직관적으로 참고할 수 있는 Swagger UI를 제공하기 위함.

## 인프라 및 배포

- **CI (지속적 통합)**: GitHub Actions
  - **선택 이유**: 코드 푸시 및 PR 생성 시 자동으로 빌드, Lint, 테스트(Vitest/Jest)를 수행하여 모노레포의 코드 품질을 방어하기 위함.
- **CD (호스팅 및 배포)**: 보류 (개발 진행 후 결정)
  - **선택 이유**: 현재는 인프라 설정보다 핵심 체스 도메인과 기능 구현에 집중하기 위해 실제 서비스 배포 환경 결정은 뒤로 미룸.

## 개발 도구 및 테스트

- **저장소 관리**: 모노레포 (Monorepo)
  - **선택 이유**: 프론트엔드(React)와 백엔드(NestJS) 간에 데이터 타입(DTO, Interface 등)을 공통 패키지에서 공유하여 API 연동 안정성과 개발 생산성을 높이기 위함.
- **패키지 매니저 및 빌드 도구**: pnpm + Turborepo
  - **선택 이유**: 빠르고 엄격한 의존성 관리(pnpm)와 효율적인 캐싱 및 병렬 빌드(Turborepo)를 통해 모노레포 개발 환경의 생산성을 극대화하기 위함.
- **테스트 도구**: Vitest (프론트/공통) + Jest (백엔드)
  - **선택 이유**: Vite와 NestJS 각각의 생태계에서 가장 기본적이고 권장되는 표준 도구를 사용하여, 모노레포 환경의 설정 복잡도와 충돌 스트레스를 최소화하기 위함.
- **코드 품질 및 포맷팅**: ESLint + Prettier (+ Husky + lint-staged)
  - **선택 이유**: 자바스크립트/타입스크립트 생태계의 절대적인 표준 도구이며, 커밋 전 자동 검사(Husky)를 통해 프로젝트 전체의 일관된 코드 스타일을 강제하고 잠재적 에러를 예방하기 위함.
