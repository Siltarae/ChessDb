# ADR-001 Game과 Repository 관계는 저장소 내부 기보 작성 시점에 연결한다

## 결정

`FEATURE-004` 저장소 관리 구현 초반에는 `Repository` 모델과 저장소 목록 조회 API를 먼저 만들고, `Game.repositoryId` 관계는 바로 추가하지 않는다.

`Game.repositoryId`는 저장소 컨텍스트 안에서 새 기보 작성 흐름을 연결하는 시점에 추가한다. 현재 계획상 우선 후보는 `TASK-035` 새 기보 작성 버튼과 입력 뷰 전환 단계다.

## 배경

`FEATURE-003`에서는 저장소 선택 UI가 없기 때문에 정식 저장 대상을 시스템 기본 저장소로 보는 임시 전제를 사용했다.

`FEATURE-004`에서는 저장소 목록, 생성, 선택, 진입 흐름을 순서대로 구현한다. 이때 `TASK-028` 저장소 목록 화면 표시는 사용자가 저장소를 볼 수 있는 진입점을 만드는 작업이며, 기보 저장 API를 특정 저장소에 귀속시키는 작업은 아니다.

## 근거

- `TASK-028`의 책임은 저장소 목록 조회와 화면 표시다.
- `Game.repositoryId`를 즉시 추가하면 기존 `POST /api/games` 저장 API, 기존 게임 저장 테스트, 시스템 기본 저장소 전제, 기존 데이터의 저장소 귀속 정책까지 함께 결정해야 한다.
- 저장소 선택 진입 전에는 프론트엔드에 현재 저장소 컨텍스트가 없다.
- 새 기보 작성이 저장소 내부에서 시작되는 시점에 `repositoryId`가 실제 요청 흐름과 직접 연결된다.
- 삭제 정책의 `deletedAt`은 `TASK-036` 저장소 삭제 확인 작업에서 추가하기로 했다.

## 영향

- `TASK-028`에서는 `Repository` 모델과 `GET /api/repositories` 목록 조회 API를 우선 구현한다.
- `TASK-031`까지는 저장소 자체의 생성과 목록 갱신에 집중한다.
- `TASK-033`에서 저장소 선택과 진입 컨텍스트를 고정한다.
- `TASK-035`에서 새 기보 작성이 선택된 저장소 안에서 시작되도록 연결하면서 `Game.repositoryId` 추가를 검토한다.
- 그 전까지 기존 기보 저장 흐름은 불필요하게 흔들지 않는다.
